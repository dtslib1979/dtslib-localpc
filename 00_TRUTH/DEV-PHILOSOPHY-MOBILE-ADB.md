# 모바일 개발 의사결정 프레임 — ADB 설계 철학

> 확정일: 2026-03-29

---

## 작업 환경 풀스택

```
[폰/태블릿]
  Termux
    └── SSH
          └── Tailscale (VPN 터널)
                └── [PC WSL2]
                      └── Claude Code
                            ├── MCP (desktop-commander / windows-cli)
                            └── Chrome (Playwright)
```

ADB = 이 구조 위에서 **폰/태블릿 화면 레이어를 PC가 건드리는 브릿지 채널**.
별도 레이어가 아니라 풀스택의 확장.

---

## 도구함 두 개 (병렬, 위아래 아님)

### ADB (구글 관할)
- **정체**: 안드로이드 OS 수준 패치 도구함. 구글이 개발자에게 열어준 원격 조종기.
- **된다**: 앱 설치/삭제/실행/종료, 파일 넣기/빼기, 화면 터치/스와이프/키입력, 스크린샷/녹화, 시스템 설정(WiFi/밝기/해상도/볼륨/DPI/애니메이션), 권한 부여/회수, 인텐트 발사, 클립보드, 로그, 프로세스 관리
- **안 된다**: 앱 내부 코드 수정. 남의 앱 안 UI/기능. Knox 보안 영역.
- **천장**: 삼성 정품 보안(Knox, 뱅킹, Samsung Pay) 유지한 채로 폰을 건드릴 수 있는 최대치.

### Good Lock (삼성 관할)
- **정체**: 삼성 UI 수준 패치 도구함. 삼성이 사용자에게 열어준 커스텀 도구.
- **된다**: 잠금화면, 알림창, 최근앱, 네비게이션바, 멀티태스킹, 키보드 테마, 테마/색상
- **안 된다**: 메뉴 구조 추가/변경. 삼성이 준 선택지 안에서만 가능.
- **천장**: 삼성이 "여기까지 바꿔도 돼" 하고 열어준 범위.

## 스텝다운 의사결정 순서

뭔가 만들어야 할 때 반드시 이 순서로:

```
1단계  Good Lock으로 되나?       → 되면 끝
2단계  ADB로 되나?               → 되면 끝
3단계  APK 만들어야 하나?        ← 여기부터 박씨 유지보수 비용 발생
4단계  PWA로 충분한가?           → 되면 설치 없이 끝
```

위에서 해결되면 밑으로 안 내려감.

## 경제 원리

| 단계 | 유지보수 주체 | 비용 |
|------|-------------|------|
| Good Lock | 삼성 | 0 |
| ADB | 구글 | 0 |
| **APK** | **박씨** | **발생** |
| PWA | 브라우저 | 0 |

APK를 최소화하는 게 이 구조의 경제 원리.

## 배제 확정 도구 (PC 없는 사람을 위한 솔루션 — 박씨 환경에 불필요)

```
Shizuku       PC 없이 폰 혼자 ADB급 권한 실행용 → PC 24/7이니 필요 없음
Tasker        폰 단독 자동화 → 터미널 공유로 대체
MacroDroid    폰 단독 매크로 → ADB input으로 대체
AutoClicker   동일 → ADB input tap으로 대체
Termux:API    대부분 → ADB가 커버 (Termux 자체는 PC 죽었을 때 백업으로 유지)
```

**박씨한테 남는 도구 딱 세 개:**
```
1. ADB        브릿지 채널 (구글 관할)
2. Good Lock  삼성 UI 커스텀 (삼성 관할)
3. APK / PWA  직접 만드는 것 (박씨 관할) — 위 둘로 안 될 때만
```

## 루팅 배제 이유

루팅 = Knox 날아감 + 뱅킹 앱 죽음 + Samsung Pay 죽음 + 보안 업데이트 끊김.
박씨는 금융 거래 필수 → 루팅은 선택지 아님.
**ADB = 보안 유지 상태에서 폰을 건드릴 수 있는 실질적 최대치.**
