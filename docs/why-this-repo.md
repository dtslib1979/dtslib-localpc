# 이 레포는 왜 존재하는가

> dtslib-localpc의 존재 이유를 파이프라인 전체에서 설명하는 문서

---

## 한 줄 요약

> **말로 시작된 아이디어가 결과물이 되려면 PC를 거쳐야 하고, 그 PC를 밖에서 쓰려고 이 레포가 존재한다.**

---

## 전체 파이프라인

```
[1] 말로 글을 쓴다 (아이디에이션)
     → 폰 + 여러 LLM (ChatGPT, Gemini, Grok, Claude)
     → 소리 듣고, 말하고, 다시 듣고 — 숙성 (채화)
     → 박씨캡처로 대화 수집 → GitHub에 텍스트로 쌓임

[2] 구조를 잡는다 (아키텍팅)
     → Claude Code + 레포에서 낙서하듯 설계
     → 수다가 커밋이 되고, 커밋이 설계 문서가 됨

[3] 만든다 (빌드)
     → 로컬 PC에서 빌드/렌더링/조립
     → GPU 연산은 외부 API, PC는 CPU 조립만

[4] 내보낸다 (배포)
     → GitHub (코드/텍스트) + YouTube (영상/음원)
     → $4/월, 로컬 저장 0 목표
```

---

## dtslib-localpc의 위치

```
입 → 폰 → LLM → 텍스트 → GitHub → Claude Code → PC에서 빌드 → 배포
                                                     ↑
                                              dtslib-localpc
                                              (이 레포가 여기 있다)
```

이 레포는 [3]번 단계를 **밖에서도 가능하게** 만드는 도구 모음이다.

---

## 두 개의 채널

아이디에이션과 아키텍팅은 다른 종류의 사고이며, 도구도 다르고, 합치면 안 된다.

| | 아이디에이션 | 아키텍팅 |
|---|---|---|
| 장소 | 밖 (이동 중) | 어디서든 (PC 접속) |
| 도구 | 폰 + 멀티 LLM | Claude Code + git |
| 속도 | 느림 (숙성 필요) | 빠름 (즉시 구현) |
| 형태 | 음성, 수다, 반복 | 텍스트, 구조, 코드 |
| 수집 | 박씨캡처 → GitHub | git commit (자동) |
| 비유 | 스케치북 | 설계도 |

---

## 왜 텍스트만으로 충분한가

모든 소재가 텍스트로 저장된다. 바이너리(영상, 음원, 이미지)는 YouTube로 빠진다.

```
GitHub Pro 5GB ÷ 소설 1권 300KB = 17,000권
10년 매일 세션 로그 = 18MB = GitHub 용량의 0.03%
```

텍스트 기반 레포지토리는 용량 걱정이 성립하지 않는다.

---

## 왜 이 대화들이 노이즈가 아닌가

헌법 1조: 레포지토리는 소설이다.

| 행위 | 자동으로 남는 것 | 콘텐츠 형태 |
|------|------------------|-------------|
| 수다 | 세션 로그, 커밋 메시지 | 에피소드 원고 |
| 문서 저장 | .md 파일 + git diff | 챕터 본문 |
| PR | 커밋 묶음 + 비교 링크 | 챕터 단위 묶음 |
| 멀티 LLM 대화 | 박씨캡처 수집본 | 크로스 LLM 비교 데이터 |

수다 → 문서 → 커밋 → PR. 별도 정리 작업 없이 작업 과정 자체가 콘텐츠화된다.
