# 리뷰: Windows/WSL/Linux 배치 백서

## 1. 잘한 점

### 프레임워크
- **3층 구조(터미널/GUI/혼종)** 는 이 대화에서 나온 가장 독창적인 인사이트다. 단순한 OS 비교가 아니라 "프로그램 형식의 시대 구분"으로 본 접근은 실용적이다.
- **"GUI 성역" 개념** — DAW, NLE, CAD를 적으로 보지 않고 필요한 영역으로 인정하는 현실적인 태도. 완전 CLI 전환을 주장하는 극단론보다 훨씬 실용적이다.

### 배치표
- 각 작업별로 Windows/WSL2/순정 Linux의 적합성을 명확히 구분함. 특히 오디오, 영상 자동화, AI/개발 영역의 구분이 정확하다.
- WSL2를 "80-90% 리눅스 대체"로 본 점은 경험적으로도 타당하다.

### FFmpeg 가설
- "영상 자동화 오케스트레이터" 전략 — NLE를 대체하려 하지 말고 FFmpeg로 반복 공정을 감싸라는 접근이 핵심이다. 이게 이 문서에서 가장 중요한 실행 원칙 중 하나다.

## 2. 문제점

### ⚠️ 인용/참조번호 [1][2][3]... — 전부 허구 (가장 큰 문제)
- 문서 전체에 [1]~[53] 번호가 붙어있으나, **이 숫자들은 실제 출처가 아니다**. 이전 세션에서 다른 모델(DeepSeek 등)이 생성한 허구 참조번호일 가능성이 높다.
- 예: [19][20][21]이 FFmpeg 부분에 붙어있지만, 실제로 존재하는 논문이나 문서를 가리키지 않음.
- 문서 자체의 논리는 타당하나, 학술적 출처가 필요한 맥락에서 사용할 수 없음.
- **권장**: 참조번호를 전부 제거하고 "일반 상식/경험적 지식"으로 표기하거나, 실제 레퍼런스를 찾아서 다시 달아야 함.

### 빠진 내용 (이 세션에서 실제로 논의된 것)
- **M5 건강검진 시스템** — 오늘 만든 2주 간격 건강검진 체계 + Telegram 봇(termius_check_bot) 인프라
- **GPU 클라우드 전략** — Vast.ai/RunPod 우선순위와 구체적 명령어 (CLAUDE.md에 상세 기재)
- **5-Lane 아키텍처** — phone_claude/phone_aider/tab_claude/tab_aider 4개 tmux 세션 체계
- **실제 WSL 이전 실행 경험** — 6/19 C→D 드라이브 이전 완료, 64.7% 여유율 확보
- **REAPER BBC SO 검증된 사실들** — Play+Record(1013)만 유효, Offline Render(42230) 무음, loopMIDI 필수 등
- **Termux 위젯 → WSL 자동기동 체인** — 리버스 터널, GatewayPorts, autossh 구조

→ 이 주제로 문서를 확장한다면 위 내용을 "부록: 실제 PC 적용 사례"로 추가하는 것이 좋다.

### 배치표의 빠진 열
- "GUI 자동화 브리지" 항목의 Windows 셀에 AutoHotkey/pywinauto가 언급되었으나, 실제로 **현재 MCP(win-gui, desktop-commander)가 이 브리지 역할을 구체적으로 수행 중**이라는 점이 빠짐.
- Docker 항목: WSL2가 Docker Desktop보다 `systemd` 내장 Docker가 더 가볍고 효율적이라는 실무 경험이 반영되지 않음.

## 3. 종합 평가

| 항목 | 평가 |
|------|------|
| 논리 구조 | ✅ 훌륭함. 3층 분류 + 배치표 구성이 명확 |
| 실용성 | ✅ 좋음. 특히 FFmpeg 오케스트레이터 전략과 GUI 성역 개념 |
| 정확성 | ✅ WSL2 범위, OS별 강점 분석이 현실과 일치 |
| **출처** | **❌ 참조번호 전부 허구. 삭제 또는 실제 출처로 교체 필수** |
| 포괄성 | ⚠️ 이 세션 M5/GPU/5-Lane/DR 내용이 누락됨. 보강 가능 |

**결론**: 문서의 논리와 결론은 타당하다. 핵심 프레임(3층 구조, GUI 성역, FFmpeg 오케스트레이터)은 유지하되, 허구 참조번호는 전부 제거하고, 이 세션에서 실제로 논의된 실행 사례(M5 건강검진, GPU 클라우드, 5-Lane)를 추가하면 더 완성도 높은 문서가 된다.
