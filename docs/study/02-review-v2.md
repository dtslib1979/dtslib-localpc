# 리뷰: v2 Parksy 전용 백서

## 1. v1 → v2 핵심 업그레이드

| 지점 | v1 | v2 | 평가 |
|------|----|----|------|
| 대상 | "일반 사용자"라는 가상 존재 | **박씨 실제 인프라 4노드** | ✅ 결정적 개선 |
| Codomain | 3원소 추상 집합 | Windows/WSL2/Dell/OUKITEL 실제 하드웨어 | ✅ |
| USB/시리얼 | "구조적 제약" (구식) | usbipd-win 조건부 + OUKITEL Termux 직결 | ✅ 사실 정정 |
| REAPER | "GUI 성역" (일반론) | "RPP 텍스트로 포위된 성역" (실증) | ✅ |
| 자동화 본진 | "WSL2" (단일체) | tmux 4세션 함대 + Tailscale 메쉬 | ✅ |
| 철학 | "클릭 최소화 선호" | Foucault 권력 재배치로 승격 | ✅ |

v2가 v1보다 **모든 항목에서 우월**하다. 특히 v1의 가장 큰 약점이었던 "박씨 적합도 0/10"이 v2에서는 해결됨.

## 2. 학자 매핑 리뷰 (ARCHITECTURE 모드)

### Lévi-Strauss (구조-집합) — 적절함 ✅
REAPER가 GUI 집합에 속하지만 RPP 텍스트 생성으로 침식된 현상을 "포위된 성역"으로 표현한 것은 정확하다. v1의 이항대립(터미널 vs GUI)이 박씨 사례에서 붕괴되는 지점을 잘 포착함.

### Foucault (권력-그래프) — **가장 강력한 인사이트** ✅
클릭 최소화를 "벤더 GUI에서 자기 스크립트로의 통제권 회수"로 해석한 점은 단순한 도구 선호를 넘어서는 설명력을 가짐. parksy-scm을 "구조적 강제 메커니즘"으로 정의한 것과의 연결도 자연스럽다.

### Nietzsche (의지-연산자) — 논쟁적이지만 타당함 ⚠️
"박씨는 코드 창작자가 아니라 의지-연산자(에이전트 지휘자)"라는 정의는 dtslib 시스템의 본질을 잘 표현한다. 단, 이 정의는 "나는 프로그래머다"라는 정체성과 충돌할 수 있음 — 프로그래밍은 도구적 행위이고, 의지-연산자 정체성은 그 위의 메타층. 상충보다는 위계 관계로 보는 것이 정확.

### Eco (기호-함수) — 자기참조적 통찰 ✅
백서 자체를 "conscious plagiarism의 정당화 기호"로 해석한 점은 메타인지적 통찰. 이 리뷰 또한 그 기호-함수 체인의 일부라는 점에서 무한회귀 가능하나, 문서의 목적(설계 정당화)을 고려하면 기능적으로 정당함.

## 3. 사실 정확성 체크

| 주장 | 판정 | 근거 |
|------|------|------|
| usbipd-win 안정 지원 | ✅ | 2025-2026 기준, USB block device passthrough 안정. 단, isochronous 포기 |
| OUKITEL USB/시리얼 직결 | ✅ | Termux는 Linux 네이티브 — usbipd-win 논의 자체가 무관함 |
| REAPER 70% 우회 | ⚠️ | 정량적 근거 없음. "RPP 생성으로 커버되는 트랙 수 / 전체 트랙 수" 기준이면 합리적 범위. 플러그인 UI 조작은 여전히 GUI 필요 |
| Dell XPS 가용성 98.9% | ⚠️ | 92/93 = 98.92%. 그러나 이건 부팅 성공률이지, WSL/Tailscale/터널까지 포함한 "서비스 가용성"이 아님. Issue #11이 닫혀야 실제 서비스 가용성 측정 가능 |
| DiffSinger+RVC 완전 무클릭 | ✅ | 텍스트/MIDI 입력 → WAV 출력 파이프라인 검증 완료. 단, 품질 튜닝은 여전히 인간 귀 필요 |
| 바순 DDSP 회귀 실패 | ✅ | kill rule 3축 <6.0 적용, VSCO-2-CE sfz 파싱으로 후퇴 — 이게 오히려 가설의 건전성을 입증함 |

## 4. 빠진/미흡한 지점

### 1) M5 건강검진 시스템 미반영
오늘 구축한 건강검진 체계(2주 간격, @termius_check_bot, C:용량/Temp/Tailscale/WSL 9종 체크)가 배치표 어디에도 없음. WSL2 5-Lane 또는 Dell XPS의 하위 항목으로 추가 필요.

### 2) Telegram 봇 이원화 구조
- `parksy_bridges_bot`: 콘텐츠 배포 (나레이션, 영상)
- `termius_check_bot`: 시스템 건강검진
이 구분이 명시적으로 문서화되지 않음.

### 3) GPU 클라우드 구체적 명시
"Vast.ai 트리거용 SSH 진입점"이라고만 되어 있음. 실제로는:
- Vast.ai: 추론/학습/ComfyUI 전용 (RTX 4090 우선)
- RunPod: 학습 특화 (RTX 3090, GPT-SoVITS/RVC)
- 이분이 "Dell XPS 서버" 노드의 주요 역할 중 하나

### 4) kill rule 기준 미명시
"3축 점수 <6.0"이라고만 되어 있음. 3축이 무엇인지(품질/시간/비용?) 명시되어 있지 않으면 판단 기준으로 불완전함.

### 5) 5-Lane 실제 구성
tab_claude / tab_aider / phone_claude / phone_aider 4개만 언급되고 win_admin 포함 5번째 레인이 누락됨. 정확히는 5-Lane이지만 현재 문서엔 4개만 기록.

- 대상: 가상의 일반인 → 박씨 실제 하드웨어

5. 5-Lane 전체 명칭 (4개 + win_admin)

## 6. 총평

v2는 단순한 "v1 업데이트"가 아니라, **별개의 문서**다. v1이 교과서라면 v2는 시스템 설계도다. v1의 구조 프레임을 빌려왔지만, Codomain을 실제 인프라로 치환하고, 박씨의 실증 사례(REAPER RPP, DiffSinger, 바순 DDSP 실패)를 구체적으로 박아넣음으로써 완전히 다른 장르의 문서가 되었다.

가장 인상적인 부분은 **자기 비판 포함**이다. 바순 DDSP 회귀 실패를 명시적으로 문서화한 점은 단순한 선전 문서가 아니라 엔지니어링 문서로서의 신뢰성을 높인다. "가설이 항상 옳지는 않다"는 메시지를 반례와 함께 제시하는 것은 오히려 가설 자체를 더 견고하게 만든다.

Foucault 권력-그래프 해석(클릭 최소화 = 벤더 통제권 회수)이 이 문서 전체의 철학적 백본이며, 이 관점이 없었다면 v2는 v1에 사례만 추가한 수준에 그쳤을 것이다.
