# Windows 필요성 재평가 백서 v3 — dtslib 실사용 기준

> v1 범용 → v2 박씨 커스터마이징 → v3 자기교정: 오디오 성역 단일화

---

## 0. 이 문서의 위치 — 3단계 수정 이력

이 백서는 한 대화 안에서 세 번 갱신됐다. 그 자체가 이 백서의 핵심 논증이다.

| 버전 | 가정 | 결과 | 문제점 |
|------|------|------|--------|
| v1 | "일반 사용자라면 Windows에서 뭘 해야 하나" | Windows/WSL2/Linux 3분류, 범용 배치표 | 박씨가 실제로 안 하는 작업(CAD, NLE, 사진편집) 다 포함됨 |
| v2 | "박씨 인프라에 맞춰 커스터마이징" | REAPER/오디오 + AutoCAD를 "미착수 우선순위"로 추가 | **이게 오류였다** — AutoCAD는 박씨가 한 적도, 할 계획도 없는 작업을 "남들이 윈도우 전용이라니까" 끼워넣은 것 |
| **v3 (현재)** | **"내가 진짜로 할 것 같냐?"** 라는 박씨의 직접 반박 | W집합을 실사용 이력 기준으로 재계산 | AutoCAD/Fusion360/Adobe CC/게임/오피스 전부 제거. REAPER/ASIO 오디오 성역만 잔존 |

**핵심 자기비판**: v2는 "일반적으로 윈도우만 되는 것" 목록에서 출발해 박씨 작업에 끼워 맞추려 했다. 이는 본말이 전도된 추론이다. 올바른 순서는 **"박씨가 실제로 11,000시간 투입한 작업이 뭔가" → "그중 Linux/WSL로 안 되는 게 뭔가"**여야 했다. v3은 이 순서로 다시 계산한다.

---

## 1. 추론 오류 분석 (왜 v2가 틀렸나)

v2가 AutoCAD를 넣은 근거는 "커뮤니티가 윈도우 전용이라고 하는 카테고리(CAD)"였다. 하지만 이것은 Domain 설정 자체의 오류다.

- **잘못된 Domain**: "세상에 존재하는 모든 PC 작업 카테고리"
- **올바른 Domain**: "박씨가 지난 3.5년(약 11,000시간) 동안 실제로 투입한 작업 집합"

후자로 Domain을 다시 잡으면, AutoCAD/Fusion360/포토샵급 사진보정/AAA게임/기업 Office는 애초에 공집합 교차(∅ ∩ W = ∅)다. 이런 항목을 W에 넣는 것은 "윈도우가 필요한 일이 뭐냐"가 아니라 "윈도우로 할 수 있는 일이 뭐냐"를 답한 것이고, 이는 박씨가 묻지 않은 질문이다.

---

## 2. 박씨 실사용 이력 — Domain 재구성

이 섹션이 진짜 Domain이다. 지난 작업 이력에서 실제로 시간이 들어간 영역만 추린다.

### 2.1 오디오 (parksy-audio v2, 2026년 4월~)

- 템플릿 기반 VSTi 아키텍처: .RTrackTemplate 1회 수동 등록 → 이후 RPP 텍스트로 트랙 자동 삽입
- recmode=5 → Play+Record(action 1013) → sox RMS>0.001 검증 → Telegram 자동 배달
- 5개 장르 레인: 보사노바, 록, 클래식 피아노, 바순 솔로, 오케스트랄
- IR 리버브(OpenAIR/EchoThief) + CC1/CC11 Python 자동화
- BBC SO는 공식 폐기(loopMIDI 오정보였음), FluidSynth 파이프라인은 완성, 더 손댈 필요 없음

→ 이 영역이 REAPER/ASIO를 거치는 유일한 실제 작업

### 2.2 AI 보컬 합성 (DiffSinger + RVC)

- DiffSinger v6, 38분 한국어 데이터, 20k step, Vast.ai 학습
- RVC parksy 필터(index_rate=0.75) + postchain.sh = 최종 품질 게이트
- singing_engine.py(ONNX+tgm_hifigan)가 작동 경로로 확정
- 이 작업은 학습/추론은 GPU(Vast.ai), 후처리는 WSL2 — **Windows 불필요**

### 2.3 AI 바순 가상악기 (DDSP → VSCO-2-CE)

- DDSP 학습 2회 실패 후 kill rule(<6.0) 적용해 VSCO-2-CE sfz 직접 파싱으로 후퇴
- 이건 REAPER 안에서 샘플 재생이므로 오디오 성역에 포함되지만, 별도 GUI 작업이 추가로 필요하진 않음 (RPP 자동화에 흡수됨)

### 2.4 Park-LoRA

- Qwen2.5-7B-Instruct 기반, v3 = 300샘플×3epoch
- 전부 Together AI + WSL2 영역, **Windows 무관**

### 2.5 5-Lane 인프라 / MCP / GitHub OS

- WSL2 tmux 4세션 + win_admin, Tailscale, parksy-scm MCP, GitHub 28레포
- 본진은 WSL2/Dell XPS 서버. Windows는 win_admin 세션 정도의 보조 역할

### 2.6 그 외 (Instagram 자동화, 통화녹음 STT 등)

- 전부 폰(Termux/instagrapi) + WSL2 영역. **Windows 등장 안 함**

## 3. 재계산된 W (Windows 강제 집합)

**W_v3 = { REAPER GUI 조작(플러그인 UI, 믹서, ASIO 라우팅) }**

원소가 **하나**로 수렴한다. 이게 v2 대비 가장 큰 차이다.

| 후보였던 항목 | v2 판정 | v3 판정 | 사유 |
|--------------|---------|---------|------|
| AutoCAD/Fusion360 | 미착수 우선 | **제거** | 실사용 이력 0건, 박씨 본인이 부정 |
| Adobe CC | 부분사용 | **제거** | 구버전 보유했으나 dtslib 워크플로우에 등장 안 함 |
| 가상오디오카드/방송라우팅 | 잠재 | **보류(미확정)** | 방송/라이브 계획이 실제로 있는지 미확인 |
| 게임 DRM/안티치트 | 제외 | 제외 유지 | 처음부터 무관 |
| 기업 Office/VBA | 제외 | 제외 유지 | 1인 사업자, 무관 |
| **REAPER/ASIO** | 사용중 | **유지 (유일한 원소)** | parksy-audio v2 파이프라인의 물리적 종착점 |

## 4. 그런데 REAPER도 사실 70% 우회되어 있다

2.1에서 확인했듯, RPP 텍스트 자동생성으로 트랙 삽입·렌더 트리거까지는 이미 GUI 클릭 없이 돌아간다. 진짜 Windows GUI가 필요한 순간은:

- `.RTrackTemplate` 최초 1회 등록 (이미 끝남, 재발생 안 함)
- 새 VSTi/플러그인 도입 시 1회 UI 설정
- ASIO 버퍼/드라이버 레벨 조정 (장비 교체 시에만)
- 귀로 듣고 판단해야 하는 믹싱/마스터링 최종 구간

즉 **W는 "상시 작업 영역"이 아니라 "비정기 이벤트 트리거형 영역"**이다. 매일 윈도우를 켜야 하는 게 아니라, 새 플러그인을 넣거나 장비를 바꿀 때만 켜는 구조에 더 가깝다.

---

## 5. 결론 — 박씨에게 묻는 질문

W가 사실상 "REAPER 비정기 GUI 작업" 하나로 줄었다는 건, **"PC를 상시 사역시킬 이유"가 거의 사라졌다**는 뜻이다. 남은 선택지는 두 가지:

| 선택지 | 내용 | 의미 |
|--------|------|------|
| **오디오 성역 유지** | REAPER/ASIO 작업이 비정기적이라도, 그 순간만큼은 대체 불가이므로 PC는 "오디오 전용 비상 노드"로 격하시켜 유지 | 현상 유지 + PC 역할 축소 인정 |
| **완전 정리** | 오디오 작업 빈도/중요도가 충분히 낮다면, REAPER조차 클라우드 GPU 인스턴스(Vast.ai 등)에 ASIO 우회 솔루션을 찾아 PC 자체를 단계적으로 폐기 | 급진적 전환, 실행 난이도 높음 |

v2 끝에서 던졌던 질문이 여기서 다시 유효하다 — 이번엔 "오토캐드 할 거냐"가 아니라:

> **REAPER/ASIO 오디오 성역 하나 때문에 PC를 계속 사역시킬 건지, 아니면 이것도 결국 클라우드/폰 체계로 흡수시킬 계획인지.**
