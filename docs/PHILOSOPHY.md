# PARKSY 인프라 철학

> 2026-03-12 대화에서 도출된 핵심 철학.
> 이 레포가 왜 존재하는지, 어디까지 쓸 수 있는지.

---

## 1. 귀찮을수록 우선순위가 높다

> "귀찮다 = 사람이 반복하기 싫다 = 정확히 자동화해야 하는 지점"

AI 에이전트(Claude Code)가 있으면 패러다임이 바뀐다.

| Before | After |
|--------|-------|
| 귀찮음 → 우선순위 낮음 → 안 함 | 귀찮음 → Claude Code한테 시킴 → 오히려 더 잘 됨 → 우선순위 높음 |

SSH 세팅 귀찮다, Telegram Bot 귀찮다, tmux 설정 귀찮다.
**그래서 이게 전부 자동화됐다.**

---

## 2. 사이즈만 커지고 엔진만 바뀐다

> "구조는 그대로, 엔진만 교체."

```
지금:    집 PC CPU  → Python 배치 → YouTube
나중에:  AWS GPU    → 같은 Python 배치 → YouTube
```

Claude Code가 자연어로 제어하는 레이어는 똑같다.
"렌더링 돌려" 하면 지금은 집 PC가 하고, 나중엔 EC2 GPU 인스턴스가 한다.

**비유:**
> 지금은 경차로 서울-부산 다니는 거고
> 나중엔 같은 도로에 트럭 올리는 거다
> 도로 설계(이 레포)는 그대로다

---

## 3. 규모별 역할 변화

| 단계 | 상황 | 이 레포 역할 |
|------|------|-------------|
| 지금 | 1인, 집 PC, 돈 없음 | 집 PC = 서버 + 자동화 전부 |
| 성장 | 콘텐츠 多, 구독자 증가 | 배치 자동화 고도화, Bot 파이프라인 확장 |
| 프로덕션 | AWS + GPU | 컨트롤 타워 — Claude Code로 인프라 제어 |
| 엔터프라이즈 | 팀 + 클라우드 | DevOps 파이프라인의 설계 기준점 |

---

## 4. 바뀌는 것 vs 안 바뀌는 것

**바뀌는 것 (하드웨어/인프라):**
- 집 PC → EC2 / GPU 서버
- 로컬 Python 배치 → Airflow / Lambda
- Telegram Bot → Slack Bot + API Gateway
- tmux → Kubernetes (극단적 확장 시)

**안 바뀌는 것 (철학/구조):**
- Claude Code로 자연어 제어하는 구조
- 귀찮은 걸 자동화한다는 철학
- 세션 로그 + 이력 관리 체계
- 훅 + 스냅샷으로 상태 추적

---

## 5. 누구한테 필요한가

| 대상 | 가치 |
|------|------|
| 1인 유튜버 / 콘텐츠 크리에이터 | 추가 비용 0원, 중소기업 수준 자동화 |
| 비개발자 Claude Code 입문자 | 삽질 기록 = 교재. 같은 혼란 겪지 않아도 됨 |
| 돈 생긴 후 클라우드 전환 시 | 설계 문서 + 마이그레이션 기준점 |
| 엔터프라이즈 | 이 레포가 레거시 자산 + 초기 MVP 레퍼런스 |
