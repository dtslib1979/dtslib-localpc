## 운영 방식

### 두 프로젝트 동시 운영

```
[PC Claude Code]     parksy-audio 렌더링 (FluidSynth + FFmpeg)
                     parksy-image Python 작업 (OpenCV, PSE 빌드)

[폰 Termux]          parksy-image Node.js 작업 (cli.js, 큐 등록)
                     경량 Git 작업
```

### RAM 사용 실측

```
Windows + 상시:       ~4.0 GB
parksy-audio 피크:    ~1.0 GB
parksy-image 피크:    ~0.4 GB
RustDesk:             ~0.1 GB
────────────────────────────
합계:                 ~5.5 GB / 16 GB
여유:                 ~10.5 GB  → 충분
```

### D: 외장 HDD 속도 문제

- USB 3.0 HDD는 NVMe 대비 30~50배 느리다
- FluidSynth SF2 로딩, WAV 쓰기가 느림
- **하지만 무인 배치라 상관없다** — 돌려놓고 나가면 됨
- 22곡 배치 ~2시간. 사람이 기다릴 일이 아님

### 이동성 유지

- D: 외장 HDD를 뽑아서 다른 PC에 꽂으면 동일 환경
- C: 드라이브는 PC마다 다르니까 작업 데이터를 C:에 두지 않는다
- 도구(Python, FFmpeg 등)만 각 PC에 설치하면 됨

### 최종 결과물 저장 전략

| 대상 | 저장소 | 비용 |
|------|--------|------|
| 코드 | GitHub | 무료 |
| 음악/영상 결과물 | YouTube 업로드 | 무료 |
| 이미지 결과물 | Google Drive | 무료 |
| D:\tmp 임시 파일 | 업로드 후 삭제 | — |

D: 용량은 1.4TB 여유. 월 ~8.5GB 증가 추정. 13년 이상 사용 가능.

## 원격 제어 전략 — 2레인 체제

> 결정일: 2026-03-01
> 핵심: 화면이 필요한 작업과 필요 없는 작업을 분리한다

### 왜 2레인인가

```
Claude Code는 1세션 = 1작업만 가능
음원 렌더링 2시간 돌리면 → 그동안 다른 거 못 함
→ 세션을 분리해서 병렬로 돌려야 한다
```

### 레인 구조

```
레인 1: CLI (Termux → GitHub → PC)
────────────────────────────────────
폰 Termux Claude Code
  → "parksy-audio 22곡 렌더링해" → git push
  → PC Claude Code가 pull → 배치 실행 (무인, 2시간)
  → 결과 git push → 폰에서 확인

용도: 음원 렌더링, 이미지 배치, 스크립트 실행
     = 화면 안 봐도 되는 작업

레인 2: GUI (RustDesk → PC 화면)
────────────────────────────────────
폰 RustDesk
  → PC 화면 직접 보면서 조작
  → APK 에뮬레이터 확인, 브라우저 테스트, OBS 설정

용도: Flutter 개발, UI 확인, 설정 변경
     = 화면을 눈으로 봐야 하는 작업
```

### PC 병렬 세션

```
PC (24시간 ON)
├── 세션 1: Claude Code CLI (터미널)
│   → 음원 배치 렌더링, 이미지 처리 등 무인 작업
│
├── 세션 2: Claude Desktop (Windows GUI)
│   → APK 개발, 코드 리뷰, 대화형 작업
│
└── 같은 Claude Pro 계정, 다른 앱 → 동시 실행 가능
```

| 항목 | 현실 |
|------|------|
| **동시 실행** | 가능. CLI와 Desktop은 별개 프로세스 |
| **같은 레포 동시 수정** | 위험. git 충돌 난다 |
| **다른 레포 각각** | 완전 안전. 이게 정답 |
| **Pro 사용량** | 두 세션 합산. 모델 선택으로 관리 |

### 인간 한계 = 2세션

3세션 이상 열어봤자 컨텍스트 스위칭 비용이 더 크다.
도구의 한계가 아니라 관리하는 사람의 한계다.
**최대 2레인이 현실적 상한선.**

## 모델 전략 — 설계와 운영의 분리

> 비싼 모델은 설계할 때만. 운영은 싼 모델로.

### 단계별 모델 배분

| 단계 | 모델 | 용도 |
|------|------|------|
| **설계** (지금) | Opus | 아키텍처, 파이프라인 설계, 복잡한 판단 |
| **운영** (나중) | Sonnet / Haiku | 이미 만들어진 스크립트 실행, 반복 작업 |

### 예시: parksy-audio

```
Opus가 할 일:     파이프라인 설계 + 렌더링 스크립트 작성 (1회)
Sonnet이 할 일:   "이 스크립트로 22곡 렌더링해" (반복)
                  → 판단 필요 없음, 실행만 하면 됨
```

### 운영 모드 최종안

```
세션 1 (Opus)     → 내가 직접 대화하면서 설계/개발
세션 2 (Sonnet)   → 만들어둔 스크립트 돌리는 무인 배치

= Opus 사용량 절약 + 병렬 작업 + 비용 최적화
```

## 향후 개선 옵션: Tailscale

> 필수 아님. 있으면 레인 1이 더 좋아진다.

```
지금:   폰 Termux → git push → PC가 pull (수동, 타이밍 갭)
개선:   폰 Termux → Tailscale VPN → PC SSH 직접 접속 (실시간)
```

| 비교 | Git 경유 (현재) | Tailscale + SSH |
|------|----------------|-----------------|
| 비용 | 0원 | 0원 (무료 티어) |
| 지연 | push/pull 갭 | 실시간 |
| 설정 | 없음 | 최초 1회 |
| PC 꺼져도 | GitHub에 남음 | 연결 끊김 |

**지금은 Git 경유로 충분. Tailscale은 여유 될 때 추가.**
